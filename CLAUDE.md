# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Critical Rules

- **NEVER edit `flake.nix` directly** — It is auto-generated by flake-file. Edit source modules and run `nix run .#write-flake` to regenerate it.
- **MUST `git add` new `.nix` files before building** — import-tree uses git tracking; untracked files are silently ignored by `nix build` and `nix run .#write-flake`.
- **Never commit `sops.key` or `keys.txt`** — these contain private decryption keys.
- **Format before committing** — run `just fmt` to apply nixfmt-rfc-style.
- **Use `btca` for up-to-date den/framework docs** — `btca ask -t den -q "..."` (see btca section below).

## Code Style

- **Formatter**: nixfmt-rfc-style (2-space indent, opening braces on same line)
- **Standard module header**: `{ inputs, den, pkgs, FTS, __findFile, ... }:` — include only what the module uses
- **Angle brackets are preferred** over dotted paths for aspect references in `includes` lists
- Use `lib.mkDefault` for values that hosts/users should be able to override; use `lib.mkForce` for invariants
- Aspect files define exactly one aspect; facet files aggregate aspects from the same category

## Architecture Overview

This is a modular NixOS (and nix-darwin) flake using **den** (declarative aspect composition), **import-tree** (automatic module discovery), and **flake-file** (generated `flake.nix`).

### Core Frameworks

| Framework | Role |
|---|---|
| **den** | Declarative aspect system — features compose into complete host/home configs |
| **import-tree** | Auto-imports every `.nix` file in `modules/`, `hosts/`, `users/` — no manual imports |
| **flake-file** | Generates `flake.nix` from `modules/flake/dendritic.nix` — never edit the output |

## Den Framework Deep Dive

### Core Concepts

**Aspects** are the fundamental unit of configuration. An aspect is an attrset with any combination of:
- `description` — human-readable string
- `nixos` — NixOS module function `{ pkgs, lib, config, ... }: { … }`
- `darwin` — nix-darwin module function (same shape)
- `homeManager` — home-manager module function
- `includes` — list of other aspects (or parametric aspect calls) to compose
- `__functor` — makes the aspect callable with parameters

**Classes** are the configuration targets: `nixos`, `darwin`, `homeManager`. An aspect can contribute to multiple classes at once.

**Namespaces** are declared in `modules/namespace.nix`:

```nix
imports = [
  (inputs.den.namespace "FTS" true)         # Feature/Tool System — exported, used everywhere
  (inputs.den.namespace "deployment" true)  # Deployment tools — exported
  (inputs.den.namespace "cody" false)       # User-specific — NOT exported
];
```

- `FTS` — all reusable feature aspects (`FTS.gaming`, `FTS.grub`, `FTS.system._.disk`, …)
- `deployment` — deployment tooling aspects
- `cody` — personal user namespace for dotfiles and overrides (`cody.dots`, `cody.fish`)

**Batteries** (`den.provides.*` / `den._.*`) are den's built-in cross-cutting providers (home-manager integration, user definition, unfree packages, etc.). They are referenced with `<den/name>` or `den._.name`.

### Context Pipeline

Den passes a **context attrset** through the composition pipeline. Aspects in `includes` lists can be plain aspect values **or** single-argument functions that receive context:

```nix
includes = [
  plainAspect                              # direct reference — no context needed
  ({ host, ... }: makeConfig host)         # receives host context
  ({ user, ... }: makeUserConfig user)     # receives user context
  ({ home, ... }: makeHomeConfig home)     # receives home context
  ({ class, ... }: { ${class}.foo = 1; })  # receives current class name
  ({ aspect-chain, ... }: takeUnused aspect-chain { … }) # receives remaining chain
];
```

Context keys available in includes:
- `host` — host attrset (`host.name`, `host.hostName`, `host.class`, `host.users`)
- `user` — user attrset (`user.userName`, `user.name`)
- `home` — home attrset (the `den.homes.*.*` entry)
- `class` — current class string (`"nixos"`, `"darwin"`, or `"homeManager"`)
- `aspect-chain` — remaining unprocessed aspects (used with `den.lib.take.*`)

Den uses `builtins.functionArgs` to detect which context keys a function needs and silently skips functions whose required keys are absent (e.g., a `{ user, ... }:` include in a host-only context is silently skipped).

### Angle Bracket Syntax

Enabled globally via `_module.args.__findFile = den.lib.__findFile` in `modules/namespace.nix`. Requires `__findFile` in the module's argument list.

| Syntax | Resolves to |
|---|---|
| `<FTS/gaming>` | `den.ful.FTS.gaming` |
| `<FTS/gaming/steam>` | `den.ful.FTS.gaming._.steam` |
| `<FTS.gaming/steam>` | `den.ful.FTS.gaming._.steam` |
| `<FTS.system/disk>` | `den.ful.FTS.system._.disk` |
| `<FTS.user/password>` | `den.ful.FTS.user._.password` |
| `<FTS.hardware._.networking/tailscale>` | `den.ful.FTS.hardware._.networking._.tailscale` |
| `<den/home-manager>` | `den.provides.home-manager` |
| `<den/primary-user>` | `den.provides.primary-user` |
| `<den/user-shell>` | `den.provides.user-shell` (callable) |
| `<den/define-user>` | `den.provides.define-user` |
| `<den.lib.parametric>` | `den.lib.parametric` (function) |

**Calling parametric aspects** — wrap with `(…)` and pass an attrset:
```nix
includes = [
  (<FTS.user/password> { method = "hashed"; value = "$6$…"; })
  (<FTS.system/disk> { type = "btrfs-impermanence"; device = "/dev/nvme0n1"; })
  (FTS.grub { uefi = true; theme = "minegrub"; })
  (<den/user-shell> "fish")
];
```

Use `den.aspects.*` directly (no angle brackets) when referring to named aspects in `den.aspects`:
```nix
includes = [ den.aspects.hm  den.aspects.hm-backup ];
```

### Aspect Patterns

#### 1. Simple Aspect

```nix
# modules/gaming/steam.nix
{ FTS, ... }:
{
  FTS.gaming._.steam = {
    description = "Steam gaming platform with performance optimizations";
    nixos = { pkgs, lib, ... }: {
      programs.steam.enable = lib.mkForce true;
      programs.gamemode.enable = lib.mkForce true;
    };
  };
}
```

**Cross-platform shorthand**: Use `os` instead of duplicate `nixos`/`darwin` blocks when both classes get identical config. Den forwards `os.*` to both `nixos.*` and `darwin.*` automatically:

```nix
FTS.coding._.cli._.just = {
  description = "Just command runner";
  os = { pkgs, ... }: {
    environment.systemPackages = [ pkgs.just ];
  };
};
```

**User-level shorthand**: Use `user` instead of `nixos.users.users.<userName>.*` in user aspects. Den forwards `user.*` → `nixos/darwin.users.users.<userName>.*` automatically:

```nix
# In den.aspects.alice or in an included aspect with user context:
den.aspects.alice.user = { pkgs, ... }: {
  packages = [ pkgs.hello ];
  isNormalUser = true;
};
```

#### 2. Facet / Aggregator

A facet has only `description` and `includes` — no `nixos`/`homeManager` blocks.

```nix
# modules/gaming/gaming.nix
{ FTS, ... }:
{
  FTS.gaming = {
    description = "All gaming platforms and tools";
    includes = [
      FTS.gaming._.steam
      FTS.gaming._.minecraft
      FTS.gaming._.lutris
    ];
  };
}
```

#### 3. Parametric via `__functor`

Used when an aspect needs caller-supplied parameters. The functor signature is:
`_self: {params}@args: { class, aspect-chain }: { includes, nixos, … }`

```nix
# modules/desktop/bootloader/grub/grub.nix
{ lib, FTS, ... }:
{
  FTS.grub.description = "GRUB boot loader configuration";
  FTS.grub.__functor =
    _self:
    { uefi ? true, devices ? [], theme ? null, ... }@args:
    { class, aspect-chain }:
    let
      themeIncludes = if theme == "minegrub" then [ FTS.grub._.themes._.minegrub ] else [];
    in
    {
      includes = themeIncludes;
      nixos = { lib, ... }: {
        boot.loader.grub.enable = true;
        boot.loader.grub.efiSupport = lib.mkForce uefi;
      };
    };
}
```

Call site: `(FTS.grub { uefi = true; theme = "minegrub"; })`

#### 4. Parametric via `den.lib.parametric`

Used when an aspect needs **context** (host/user info) but no caller parameters. `den.lib.parametric` marks the aspect so den passes context to its `includes` functions.

```nix
# modules/aspects/hostname.nix
{ den, lib, FTS, ... }:
{
  FTS.hostname = den.lib.parametric {
    description = "Set hostname from den host context";
    includes = [
      ({ host, ... }: {
        ${host.class}.networking.hostName = lib.mkDefault (host.hostName or host.name);
      })
    ];
  };
}
```

#### 5. Combined `__functor` + `den.lib.parametric`

Used when an aspect needs **both** caller parameters and context injection. The `__functor` processes caller args, then returns `<den.lib.parametric> { … }`.

```nix
# modules/system/password.nix
{ den, lib, FTS, __findFile, ... }:
{
  FTS.user._.password.__functor =
    _self: arg:
    let config = mkPasswordConfig arg;
    in
    <den.lib.parametric> {
      description = "System password configuration";
      includes = [
        ({ user, ... }: { nixos.users.users.${user.userName}.hashedPassword = config.value; })
        ({ host, ... }: lib.mkIf (host.users == {}) { nixos.users.users.nixos.initialPassword = "nixos"; })
      ];
    };
}
```

Call site: `(<FTS.user/password> { method = "hashed"; value = "$6$…"; })`

#### 6. Direct `den.aspects.*`

Named aspects in the `den.aspects` namespace are defined directly and referenced by dotted path (no angle brackets). Used for built-in den features and user/host-specific named aspects.

```nix
# In a user or host file:
den.aspects.cody = {
  description = "Cody user configuration";
  includes = [ … ];
  homeManager = { … }: { … };
};
```

Reference: `den.aspects.cody` or `den.aspects.hm-backup`

#### 7. Private Namespace Aspect

The `cody` namespace is not exported and is used for personal dotfiles and user-specific overrides.

```nix
# users/cody/dots/dots.nix
{ … }:
{
  cody.dots = {
    homeManager = { config, pkgs, ... }: {
      home.file.".config/ghostty".source = dotsLink "config/ghostty";
    };
  };
}
```

Reference in includes: `cody.dots` (plain dotted path, no angle brackets needed).

## Host Declaration

Hosts live in `hosts/<hostname>/<hostname>.nix`. A host file has two parts:

```nix
# hosts/THEBATTLESHIP/THEBATTLESHIP.nix
{ inputs, FTS, __findFile, ... }:
{
  # ── Part 1: Register the host with den ──────────────────────────────────
  den.hosts.x86_64-linux = {
    THEBATTLESHIP = {
      description = "Main desktop system";
      users.cody = {};          # Associates the cody home with this host
      aspect = "THEBATTLESHIP"; # Which den.aspects.* entry to use
    };
  };

  # ── Part 2: Define the host aspect ──────────────────────────────────────
  den.aspects.THEBATTLESHIP = {
    includes = [
      <FTS/fonts>
      <FTS/gaming>
      (FTS.grub { uefi = true; })
      (<FTS.system/disk> { type = "btrfs-impermanence"; device = "/dev/nvme0n1"; })
      (<FTS.user/password> { method = "hashed"; value = "$6$…"; })
      (<FTS.deployment> {})
    ];

    # Inline NixOS config for host-specific settings that don't belong in an aspect
    nixos = { config, lib, pkgs, ... }: {
      time.timeZone = "America/Los_Angeles";
      boot.loader.grub.configurationLimit = 15;
      nix.gc = { automatic = true; dates = "weekly"; options = "--delete-older-than 7d"; };
    };
  };
}
```

- `den.hosts.<system>.<NAME>` — registers the host; `system` is `x86_64-linux`, `aarch64-linux`, `aarch64-darwin`, etc.
- `users.<username> = {}` — each entry creates an association so the user's home is built alongside the host.
- `aspect = "NAME"` — tells den which `den.aspects.NAME` attrset to use as the top-level aspect.

## User / Home Declaration

Users live in `users/<username>/<username>.nix`. A user file has two parts:

```nix
# users/cody/cody.nix
{ FTS, den, cody, __findFile, ... }:
{
  # ── Part 1: Register homes with den ─────────────────────────────────────
  den.homes = {
    aarch64-darwin.cody = { userName = "CodyWright"; aspect = "cody"; };
    x86_64-linux.cody   = { userName = "cody";       aspect = "cody"; };
  };

  # ── Part 2: Define the user aspect ──────────────────────────────────────
  den.aspects.cody = {
    description = "Cody user configuration";

    includes = [
      den.aspects.hm-backup          # Built-in den aspect (use den.aspects.* syntax)
      <FTS.apps/browsers>            # Angle bracket FTS aspect
      <FTS.coding/editors>
      <den/primary-user>             # Battery: marks this as the primary user
      (<den/user-shell> "fish")      # Battery: sets fish as default shell
      cody.dots                      # Private namespace aspect (dotfiles)
      cody.fish                      # Private namespace aspect (fish config)
      FTS.apple-fonts                # Direct dotted reference (also valid)
    ];

    # Inline home-manager config for things that don't belong in a separate aspect
    homeManager = { ... }: {
      programs.ssh.enable = true;
      programs.ssh.matchBlocks."starcommand" = { hostname = "192.168.0.102"; user = "root"; };
    };
  };
}
```

- `den.homes.<system>.<username>` — registers a home; `userName` is the actual Unix username.
- Multiple systems can share the same `aspect` name (e.g., both darwin and linux use `"cody"`).
- The `cody` namespace argument must be listed in the module args to access `cody.*` aspects.

## Global Defaults

Every host and home automatically includes the aspects defined in `modules/aspects/defaults.nix`:

```nix
# modules/aspects/defaults.nix
{ __findFile, den, ... }:
{
  den.default = {
    includes = [
      <den/home-manager>    # home-manager integration battery
      <den/define-user>     # user definition battery
      den.aspects.hm        # home-manager CLI helper
      den._.inputs'         # makes flake inputs available as inputs'
      den._.self'           # makes flake self available as self'
      <FTS/base-host>       # base host configuration
      <FTS/nix-settings>    # nix daemon settings (substituters, trusted users, …)
      <FTS/state-version>   # NixOS state version tracking
      <FTS/hostname>        # sets networking.hostName from den context
      <FTS/no-man-cache>    # disables man-db caching (speeds up builds)
    ];

    home.includes = [
      <FTS/nix>             # nix CLI settings for home-manager
      <FTS/user-secrets>    # SOPS user secret management
    ];
  };
}
```

You do **not** need to re-include these in host or user aspects — they are automatic.

## Adding New Aspects

1. Create the file (import-tree picks it up automatically — no manual imports):
   ```
   modules/<category>/<name>.nix
   ```

2. Define the aspect:
   ```nix
   { FTS, ... }:
   {
     FTS.<category>._.<name> = {
       description = "What this does";
       nixos = { pkgs, lib, ... }: { /* NixOS config */ };
       homeManager = { pkgs, ... }: { /* home-manager config */ };
       includes = [ <FTS/other/aspect> ];
     };
   }
   ```

3. **`git add` the new file** before building — import-tree only tracks git-staged files:
   ```bash
   git add modules/<category>/<name>.nix
   ```

4. Reference it in a host or user's `includes`:
   ```nix
   includes = [ <FTS.<category>/<name>> ];
   ```

5. Build and test: `just build-host <host>` or `just test`.

## Adding Flake Inputs

`flake.nix` is generated — never edit it directly. To add a new input:

1. Open `modules/flake/dendritic.nix` and add the input declaration:
   ```nix
   flake-file.inputs.<input-name>.url = lib.mkDefault "github:owner/repo";
   flake-file.inputs.<input-name>.inputs.nixpkgs.follows = "nixpkgs"; # if applicable
   ```

2. Regenerate `flake.nix`:
   ```bash
   nix run .#write-flake
   ```

3. `git add flake.nix flake.lock modules/flake/dendritic.nix` and commit.

4. Access the input in modules via the standard `inputs.<input-name>` argument.

## den.provides (Batteries) Reference

Batteries are den's built-in cross-cutting providers. Access them via `<den/name>` angle brackets or `den.provides.<name>` / `den._.<name>`.

| Battery | Syntax | Purpose |
|---|---|---|
| `home-manager` | `<den/home-manager>` | Integrates home-manager into NixOS |
| `define-user` | `<den/define-user>` | Creates Unix users declared in `den.hosts.*.users` |
| `primary-user` | `<den/primary-user>` | Marks a home as the primary user (first in list) |
| `user-shell` | `(<den/user-shell> "fish")` | Sets default shell; enables it on OS + home |
| `unfree` | `(<den/unfree> true)` or `(<den/unfree> ["steam"])` | Allows unfree packages globally or by name |
| `hostname` | via `den._.` | Hostname battery (usually via `<FTS/hostname>`) |
| `inputs'` | `den._.inputs'` | Exposes flake inputs as `inputs'` in modules |
| `self'` | `den._.self'` | Exposes flake self as `self'` in modules |
| `mutual-provider` | Den internals | Coordinates mutual dependencies |
| `forward` | Den internals | Forwards aspects to another context |

## Secrets Management (SOPS)

- Host secrets: `hosts/<hostname>/secrets.yaml`
- User secrets: `users/<username>/secrets.yaml`
- Edit with: `just edit-secrets <name>` (auto-detects host vs user)
- Keys: `hosts/<hostname>/facter.json` contains the host SSH key used for SOPS decryption
- The SOPS age key is derived from `/etc/ssh/ssh_host_ed25519_key`
- **Never commit**: `sops.key`, `keys.txt`, or any unencrypted secret

In NixOS modules, access secrets via `config.sops.secrets."path/to/secret".path`.

## Common Pitfalls

1. **Forgetting `git add` before building**
   import-tree tracks files via git. A new `.nix` file that is not staged is completely invisible to `nix build`, `nix run .#write-flake`, and `just test`. Always `git add <file>` immediately after creating it.

2. **Editing `flake.nix` directly**
   The file is overwritten by `nix run .#write-flake`. All changes are lost. Edit `modules/flake/dendritic.nix` instead.

3. **Missing `__findFile` in module args**
   Angle bracket syntax (`<FTS/foo>`) requires `__findFile` to be in the module's argument list. If you see "undefined variable `__FTS`" or similar errors, add `__findFile` to the args.

4. **Forgetting `{ class, aspect-chain }:` in `__functor` return**
   A `__functor` must return a function that accepts `{ class, aspect-chain }`. Omitting this makes the aspect non-callable by den's composition engine.

5. **Using angle brackets for `den.aspects.*`**
   `<den/home-manager>` works (resolves to `den.provides.home-manager`), but `den.aspects.cody` and `den.aspects.THEBATTLESHIP` are accessed by dotted path — they are not in `den.provides`.

6. **Wrong context key**
   Use `{ host, ... }:` for host-level config (networking, filesystem), `{ user, ... }:` for per-user config (passwords, shells), and `{ home, ... }:` for home-manager config. Den silently skips includes whose required context keys don't exist in the current pipeline stage.

7. **`lib.mkForce` vs `lib.mkDefault` confusion**
   Use `lib.mkDefault` in aspects/modules that should be overridable by hosts. Use `lib.mkForce` for invariants that must not be changed. Forgetting `lib.mkForce` on conflicting options causes evaluation errors.
